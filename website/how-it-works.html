How it works

The safety graph, explained

Safe Walk builds a scored safety layer on top of Google Maps by combining municipal open data, community reports, and time-aware scoring into a single API that any platform can query.

Architecture

When you request a route, Safe Walk calls the Google Routes API with computeAlternativeRoutes: true to get up to three different walking paths between your origin and destination. Each path comes back as an encoded polyline, which is a compressed list of latitude/longitude points along the route.

We decode each polyline, sample points along the route, and score every sampled point against our safety graph. The route with the lowest average danger score becomes the "safest" recommendation. The route Google ranks first (optimized for speed) is labeled "fastest." You see both.

The safety scoring engine

Every point on every route gets a safety cost between 0 (very safe) and 1 (very dangerous). The cost is computed from three signals:

Incident density

We query all police incidents within 500 meters of the point. Each incident's contribution is weighted by three factors:

Distance weight = 1 / (1 + meters / 100)

An incident 50m away contributes 3x more than one at 300m.

Recency weight = e^(-days_old / 365)

Last month's incident counts heavily. Two years ago is nearly zero.

Severity weight

Assault, Robbery, Sexual Violation = 1.0 | Break and Enter = 0.5 | Theft = 0.3 | Auto Theft = 0.2

Incident contribution = distance x recency x severity

Streetlight coverage

We count streetlights within 100 meters of each point. Five or more nearby streetlights means the area is well-lit and the lighting danger score drops to zero. Zero streetlights means maximum lighting danger. This layer is weighted heavily after dark and almost ignored during daytime.

Time-of-day adjustment

The scoring engine knows what time it is. During the day (7 AM to 7 PM), incident history carries most of the weight. After dark (9 PM to 5 AM), streetlight coverage becomes much more important. Dawn and dusk hours interpolate smoothly between the two modes. The transition follows Toronto's actual patterns.

The final score

safety_cost = incident_score x incident_weight + lighting_score x lighting_weight

Clamped between 0.0 (safe) and 1.0 (dangerous)

Night weights: incident = 0.6, lighting = 0.4

Day weights: incident = 0.8, lighting = 0.1

The route's overall safety score is the average of all sampled points along the path. A route scoring 0.15 is meaningfully safer than one scoring 0.40, even though both numbers are below 0.5. The comparison between routes is what matters, and when you see both routes on the map, the difference is usually obvious: the safe route sticks to well-lit main streets while the fast route cuts through darker, higher-incident areas.

Data sources

Toronto Police Service, Major Crime Indicators

Every geocoded assault, robbery, break-and-enter, theft, and sexual violation. Updated regularly via the ArcGIS feature service at data.torontopolice.on.ca. Tens of thousands of records covering multiple years.

City of Toronto, Street Lighting

The location of every streetlight in the city, approximately 160,000 points. Sourced from Toronto's Open Data portal via CKAN API. Includes type and wattage where available.

Community reports

User-submitted reports of harassment, poor lighting, suspicious activity, and other safety concerns. Weighted lower than police data initially, but gain weight with corroboration from multiple users.

Google Routes API

Walking route geometry and alternatives. We use Google for what it's good at (pathfinding on the road network) and add what it's missing (safety scoring).

API reference

Safe Walk exposes two main endpoints. Both return JSON.

POST /api/route

Returns the fastest and safest walking routes between two addresses.

// Request
POST /api/route
{ "origin": "string", "destination": "string" }

// Response
{
  "fastest": {
    "polyline": "encoded_polyline_string",
    "distanceMeters": 1200,
    "duration": "900s",
    "safety_score": 0.42
  },
  "safest": {
    "polyline": "encoded_polyline_string",
    "distanceMeters": 1580,
    "duration": "1140s",
    "safety_score": 0.18
  },
  "all_routes": [ ... ]
}

GET /api/safety-score

Returns the safety cost for a single point at a given hour.

// Request
GET /api/safety-score?lat=43.6520&lng=-79.3718&hour=23

// Response
{ "safety_cost": 0.67, "lat": 43.652, "lng": -79.3718, "hour": 23 }

What's next

The current scoring engine uses statistical weighting. The next version adds an AI prediction layer using Claude to estimate risk in areas with sparse police data, based on patterns from nearby areas: time clustering, proximity to certain location types, and historical incident trajectories.

We're also expanding the community reporting layer and building integrations for university campus safety apps, starting with University of Toronto, TMU, and York.
